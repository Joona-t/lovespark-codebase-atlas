use clap::{Parser, Subcommand};
use std::path::PathBuf;

/// Real-time character-grid image renderer.
///
/// Default behavior with no subcommand: launches the visual studio (GUI).
/// Subcommands provide headless operation for scripts and AI agents.
#[derive(Parser, Debug)]
#[command(name = "glyph-grid-studio", version, about, long_about = None)]
struct Cli {
    #[command(subcommand)]
    command: Option<Command>,
}

#[derive(Subcommand, Debug)]
enum Command {
    /// Launch the visual studio (default if no subcommand given).
    Studio,

    /// Render a single image to an animated GIF, headlessly.
    Render(RenderArgs),

    /// Render many variants of the SAME source in ONE Tauri session.  Reads a
    /// manifest JSON with `{in, frames, jobs:[{name,out,format?,config}, ...]}`
    /// and runs all jobs back-to-back without re-launching the app per variant.
    /// ~10× faster than calling `render` repeatedly because Tauri startup +
    /// source-image decode happens once.  See ITER-024 in BUGS_AND_ITERATIONS.md.
    Batch(BatchArgs),

    /// Print the catalog of available palettes / ramps / dithers / glyph sets / postprocess stages.
    Catalog,

    /// Run the MCP server (Model Context Protocol over stdio JSON-RPC).
    /// Wire this into Claude Desktop / Cursor via their MCP config; tools then
    /// appear as `glyph_grid_render` and `glyph_grid_catalog`.
    Mcp,
}

The Naive Version

🤖 LLM-drafted from deterministic signal · override via audit-meta.yml

A single Python script: PIL to load the image, downsample to a character grid, map each cell's brightness to a glyph from a density ramp, print to stdout. Maybe 40 lines. The GUI, the Tauri shell, the MCP server, the Rust — all of that is scaffolding around `" .:-=+*#%@"[int(brightness * 9)]`.

The Assumptions

🤖 LLM-drafted from deterministic signal · override via audit-meta.yml
  • The OS always has fonts available — character-grid rendering silently assumes the chosen glyph font is installed and metrics-stable; on a fresh macOS user account or a stripped-down environment the fallback font has different advance widths and the grid stops being a grid.
  • p5.js finishes drawing before the screenshot/MCP call returns — there's no explicit signal in "real-time renderer + MCP server" that the Rust side waits for a frame to actually commit, so naively the MCP caller gets whatever pixels happened to be in the canvas at poll time.
  • The input image fits in memory and decodes successfully — no declared deps means no image library is named, so the code is assuming whatever Tauri/webview hands it is already a valid bitmap of reasonable size. A 50MP HEIC or a corrupt PNG isn't on the radar.
  • Tauri's IPC channel between JS and Rust is lossless and ordered — 174 files and a GUI+CLI+MCP trifecta usually means three callers racing for the same renderer; you should be nervous about whether two concurrent MCP calls clobber each other's canvas state.
  • Honestly, I can't tell from deps+permissions alone what filesystem/network surface this has — "top deps: none declared" is suspicious for a 48k LOC Tauri app and probably means the Rust `Cargo.toml` wasn't sampled, so any assumption about sandboxing, file-dialog scope, or MCP transport (stdio vs socket) is one I'd refuse to make without reading `src-tauri/`.

The Decisions

🤖 LLM-drafted from deterministic signal · override via audit-meta.yml

- 2026-04 — Vendored p5.js into Tauri 2 instead of a web app: the WHY is that this is a macOS-targeted real-time renderer, and a browser tab can't give you GPU-stable frame pacing, a CLI, and an MCP server from one codebase. Honestly the naive thing would be a static p5.js page on a domain; that gets you a demo, not a tool you can script against from another agent. - 2026-05 — Per-effect LUTs and persistent buffers (linearToSrgb 1024-entry LUT, vignette LUT, chromatic/barrel persistent buffers): the WHY is that per-pixel math inside a JS render loop is where frame time goes to die, and the opt-100/200 commits show real numbers (−50.97ms/frame geomean on one of them). You should be nervous about a 174-file project with zero declared deps trying to hit 60fps without precomputed tables. - 2026-05 — A BUGS_AND_ITERATIONS-style log (ITER-100/101/103, LOOP-BUG-001, opt-NNN commit prefixes): the WHY is that perf work without a written trail turns into vibes-driven optimization where you can't tell which change bought which millisecond. The numbered ledger forces each commit to claim a measurement, which is the only way the next session knows what's already been tried.

The Trade-offs

🤖 LLM-drafted from deterministic signal · override via audit-meta.yml

- We chose Tauri + Rust over Electron because the binary stays small and native, but lost the ability to share a single JS codebase across desktop and web — the renderer is p5.js, the shell is Rust, and any new platform feature crosses a language boundary. - We chose p5.js as the rendering core over a custom WebGL/Canvas engine because it ships fast and the glyph-grid model maps cleanly onto p5's draw loop, but you should be nervous about real-time performance on large grids — p5 is not built for tight per-frame budgets and 48k LOC of sketch code won't outrun that. - We chose GUI + CLI + MCP server as three surfaces over picking one because it lets the same renderer get driven by a human, a shell pipeline, or an agent, but tripled the integration surface — every config change has to be threaded through all three entry points or they drift. - I can't tell from this context (no declared deps, no audit tldr) how the p5.js sketch is actually wired into Tauri's webview, so I can't honestly score the JS↔Rust boundary trade-off beyond "it exists."

Silent failure modes

🤖 LLM-drafted from deterministic signal · override via audit-meta.yml

- If a Tauri IPC handler silently drops oversized image payloads above the default 1MB limit, the CLI/MCP path keeps returning "success" with a truncated or empty grid and no error surfaces to the caller. - If p5.js loadImage() resolves before pixel data is actually populated (the classic race), renders save a blank or partially-filled canvas that looks plausible at thumbnail size but is wrong at full resolution. - If glyph density lookup tables drift out of sync between the Rust CLI path and the JS GUI path, the same input image produces visually-similar-but-different output across surfaces, and nobody notices until a user diffs two renders. - If the MCP server caches rendered grids by input path without hashing file contents, edits to the source image return stale output indefinitely — the file mtime check that "should" exist is the kind of thing that gets stubbed during dev and never revisited. - If Tauri's webview throttles requestAnimationFrame when the window is backgrounded, long batch renders kicked off from the MCP server complete in 10x the expected wall time with no log line indicating why, and users assume the tool is broken rather than minimized.
